# How to Create a Skill — Step-by-Step Guide

This guide walks you through creating a professional skill from scratch using the `SKILL.md` skeleton provided in this folder.

---

## Before You Start

- [ ] You have identified the specific problem your skill solves
- [ ] You have tested all commands/scripts locally
- [ ] You understand who will use this skill and why
- [ ] You have a clear name in kebab-case (e.g., `markdown-to-pdf`, `github-pr-manager`)

---

## Step 1: Set Up the Directory Structure

Copy the template to `skills/` with your skill name:

```bash
cd ../skills/
cp -r ../TEMPLATE/template-skill/ my-skill-name/
cd my-skill-name/
```

This gives you:

- `SKILL.md` — Ready to customize (the skeleton)
- `scripts/` — For your code (with `__init__.py` for imports)
- `tests/` — For unit/integration tests (with `__init__.py`)
- `assets/`, `references/` — For resources
- `requirements.txt` — For Python dependencies
- `pytest.ini` — For test configuration
- `.gitignore` — To exclude common unwanted files
- `example.py`, `sample-data.json`, `EXAMPLE-API.md` — Reference files to customize

You also have access to `../TEMPLATE/HOW-TO-CREATE-A-SKILL.md` as reference.

---

## Step 2: Fill in the YAML Frontmatter

Open `my-skill-name/SKILL.md` and update the frontmatter block:

```yaml
---
name: my-skill-name # kebab-case, unique
version: 1.0.0 # semantic versioning
author: Your Name # who created this
description: "Brief description: what problem it solves, when to use it (1-2 sentences)"
argument-hint: "[primary-arg] [optional-args]" # optional: how to invoke
user-invocable: true # true if users call it directly
applyTo: "**" # optional: pattern for auto-activation
---
```

**Rules**:

- `name`: Use lowercase with hyphens only. Must be unique within this repository.
- `version`: Start at `1.0.0`. Increment: `1.0.1` (patch), `1.1.0` (minor), `2.0.0` (major).
- `description`: One sentence describing the problem + one sentence on when to use it.
- `argument-hint`: Show the command syntax concisely (e.g., `"[file.md] [--output-dir]"`).

## Step 3: Update the H1 Title

Change:

```markdown

```

To:

```markdown
# [Your Skill Name in Title Case]
```

Examples:

- `# Markdown to PDF Converter`
- `# GitHub PR Manager`
- `# PDF Batch Processor`

## Step 4: Write "When to Use"

Replace the placeholder with **concrete, specific scenarios**:

```markdown
## When to Use

Activate this skill when the user:

- Needs to [specific action 1]
- Wants to [specific action 2]
- Requires [specific action 3]
- Has [specific problem 4]
```

**Rules**:

- Minimum 3 bullet points, maximum 5.
- Each bullet must describe a real user need, not a feature.
- Use simple, direct language: "needs", "wants", "requires", "has".

**Example (Good ✅)**:

```markdown
- Needs to convert a Markdown file to a formatted PDF
- Wants to process multiple .md files with consistent styling
- Requires PDF metadata (title, author) to be auto-populated
- Has a file to preview before distributing
```

**Example (Bad ❌)**:

```markdown
- PDF conversion
- Batch processing
- Styling support
```

---

## Step 5: Define the Output

Describe **exactly** what your skill produces:

```markdown
## Output

This skill produces:

1. **[Output type 1]** — [Concrete description: what is it, format, where is it saved]
2. **[Output type 2]** — [Concrete description]
3. **[Output type 3]** (optional) — [Concrete description]
```

**Rules**:

- Use concrete terms: "PDF file", "JSON report", "CSV data", "modified Python file".
- Include format and location: "generates `output.pdf` in the same directory".
- Maximum 3 outputs; if more, the skill is doing too much.

**Example (Good ✅)**:

```markdown
1. **PDF file** — `output.pdf` in the same directory, fully formatted with embedded fonts and images
2. **Metadata JSON** (optional) — `output.metadata.json` with title, author, page count, creation date
3. **HTML preview** (optional) — `output.html` generated by wkhtmltopdf, opens in browser automatically
```

**Example (Bad ❌)**:

```markdown
1. A file
2. Some output
3. Results
```

## Step 6: List Prerequisites

Be **explicit and complete**. Include installation instructions for all major OSes.

````markdown
## Prerequisites

### Required

- **[Tool/Language 1]** version [X.Y]+ — [What it does]
- **[Tool/Language 2]** — [What it does]
- **[Credential/API]** — [Where to obtain, how to get]

### Installation

**macOS**:

```bash
brew install [package]
```
````

**Ubuntu/Debian**:

```bash
sudo apt install [package]
```

**Windows**:

```bash
# Via package manager or manual
[installation steps]
```

### Verify

```bash
[verification command]
```

Expected output:

```
[what success looks like]
```

````

**Rules**:
- List EVERY dependency explicitly (not just "Node.js"; say "Node.js 16+").
- Provide installation for macOS, Ubuntu/Debian, and Windows.
- Include a verification step that confirms installation.
- Show exact expected output.

**Example (Good ✅)**:
```markdown
### Required

- **Python** 3.8+ — Language runtime
- **pandoc** — Document converter
- **wkhtmltopdf** — HTML to PDF rendering engine

### Installation

**macOS**:
```bash
brew install python pandoc wkhtmltopdf
````

**Ubuntu/Debian**:

```bash
sudo apt update && sudo apt install python3 pandoc wkhtmltopdf
```

**Windows** (via Chocolatey):

```bash
choco install python pandoc wkhtmltopdf
```

### Verify

```bash
python --version  # Output: Python 3.8.x or higher
pandoc --version  # Output: pandoc X.Y.Z
wkhtmltopdf --version  # Output: wkhtmltopdf X.Y.Z
```

````

## Step 7: Write Procedure Steps

Each step must be:
1. **Self-contained**: reader can follow without external knowledge
2. **Testable**: actual, working commands
3. **Explicit**: includes expected output
4. **Concrete**: no placeholders like `[your-file]` without examples

```markdown
## Procedure

### Step 1: [What This Step Does]

[1-2 sentences explaining the goal.]

```bash
[Real, working command with actual file names/examples]
````

**Output**:

```
[Exact, word-for-word output or confirmation message]
```

### Step 2: [What This Step Does]

[1-2 sentences.]

```bash
[Real, working command with options]
```

**Options**:

- `--flag` — [What it does]
- `--other-flag` — [What it does]

**Output**:

```
[Exact output]
```

````

**Rules**:
- Number steps sequentially (## Step 1:, ## Step 2:, etc.)
- Each step must be copyable and immediately executable.
- Show realistic command examples, not pseudocode.
- Describe what output the user should see.
- Document all command-line flags and options.

**Example (Good ✅)**:
```markdown
### Step 1: Convert a Single File

Convert a Markdown file to PDF with default styling.

```bash
python scripts/markdown_to_pdf.py README.md
````

**Output**:

```
Converting README.md to PDF...
✅ Conversion complete!
📄 Output: README.pdf (1.2 MB, 5 pages)
```

````

## Step 8: Add Examples

Provide **at least 2 real-world examples**: one simple, one advanced.

```markdown
## Examples

### Example 1: [Simple, Common Scenario]

**User Request**: "[What the user asks for]"

```bash
[Real, executable command]
````

**Result**:

- [Concrete outcome 1]
- [Concrete outcome 2]

---

### Example 2: [Advanced, Complex Scenario]

**User Request**: "[What the user asks for]"

```bash
[Real, executable command with multiple options]
```

**Result**:

- [Concrete outcome 1]
- [Concrete outcome 2]
- [Concrete outcome 3]

````

**Rules**:
- Frame examples as real user requests ("I want to...", "I need to...").
- Commands must be copy-pasteable and actually work.
- Show both what the user does and what they get.
- Example 1 should take <1 minute; Example 2 should showcase advanced features.

**Example (Good ✅)**:
```markdown
### Example 1: Convert a Single Markdown File

**User Request**: "I have a README.md file and I want to convert it to PDF"

```bash
python scripts/markdown_to_pdf.py README.md
````

**Result**:

- `README.pdf` created in the same directory
- File includes embedded images, syntax-highlighted code blocks
- Ready to share or print

---

### Example 2: Batch Convert with Custom Styling and Metadata

**User Request**: "Convert all my documentation files to PDF with academic formatting and add metadata"

```bash
python scripts/batch_convert.py ./docs/ --style academic --output ./pdfs/ --metadata --author "Jane Doe"
```

**Result**:

- All `.md` files in `./docs/` converted to PDF
- Saved to `./pdfs/` with descriptive names
- Academic styling applied (serif fonts, formal spacing)
- Metadata added: author name on all files
- Processing time: ~2 minutes for 20 files

## Step 9: Document Reference & FAQ

Add a reference table if your skill has multiple commands or options:

```markdown
## Reference

| Command | Purpose | Input | Output |
| ------- | ------- | ----- | ------ |
| `python scripts/cmd1.py` | [Purpose] | `input.txt` | `output.txt` |
| `python scripts/cmd2.py` | [Purpose] | `folder/` | `results/` |
````

Add FAQ for common questions:

```markdown
## FAQ

**Q: Can I use this with .docx files?**  
A: No, this skill is optimized for Markdown. Use pandoc directly for other formats: `pandoc input.docx -o output.pdf`

**Q: How do I customize colors in the PDF?**  
A: Edit the CSS files in `assets/styles/` or create a new style in `scripts/apply_style.py`

**Q: What's the maximum file size?**  
A: Tested up to 50MB (≈2000 pages). Larger files may require splitting.
```

## Step 10: Add Troubleshooting

Document **at least 3 common errors** users might encounter:

````markdown
## Troubleshooting

### Issue: "pandoc: command not found"

**Cause**: pandoc is not installed or not in your system PATH.

**Solution**:

```bash
# Verify pandoc is installed
which pandoc  # macOS/Linux
where pandoc  # Windows

# If not found, reinstall
brew install pandoc  # macOS
sudo apt install pandoc  # Ubuntu/Debian
```
````

### Issue: "No such file or directory: input.md"

**Cause**: The file path is incorrect or the file doesn't exist in that location.

**Solution**:

```bash
# Check that the file exists
ls input.md  # macOS/Linux
dir input.md  # Windows

# Use absolute path if needed
python scripts/markdown_to_pdf.py /full/path/to/input.md
```

### Issue: "Output file is blank or corrupted"

**Cause**: The Markdown file contains unsupported syntax or wkhtmltopdf encountered a rendering error.

**Solution**:

```bash
# Test with a simple Markdown file first
echo "# Hello\n\nThis is a test." > test.md
python scripts/markdown_to_pdf.py test.md

# If that works, check your input file for complex elements
```

````

**Rules**:
- Document errors users are likely to encounter.
- Explain the root cause.
- Provide step-by-step solution.
- Include verification commands.

## Step 11: Create Scripts

Place all executable code in `scripts/`:

```bash
scripts/
├── main_script.py        # Primary logic
├── helper.py             # Supporting functions
└── requirements.txt      # Python dependencies
````

**If using Python**:

```bash
cd my-skill-name
pip install -r scripts/requirements.txt
python scripts/main_script.py input.txt
```

Make sure scripts are **tested and working** before documenting them.

## Step 12: Final Verification Checklist

Before publishing, verify all of the following:

### Content

- [ ] Frontmatter YAML is complete and valid
- [ ] H1 title matches the skill purpose
- [ ] "When to Use" has 3+ concrete bullet points (no vague language)
- [ ] "Output" clearly describes what is produced (format, location, size)
- [ ] "Prerequisites" includes installation steps for macOS, Ubuntu/Debian, and Windows
- [ ] "Procedure" steps are numbered and fully executable
- [ ] Each command includes expected output
- [ ] At least 2 Examples (simple + advanced)
- [ ] At least 3 Troubleshooting issues documented
- [ ] FAQ section addresses common questions

### Testing

- [ ] I have tested EVERY command locally on my machine
- [ ] All file paths in commands are correct
- [ ] All commands produce the expected output shown in the documentation
- [ ] Prerequisites verification steps work correctly
- [ ] I have tested at least one Example end-to-end

### Files

- [ ] `SKILL.md` exists and is complete
- [ ] `scripts/` folder contains all code
- [ ] `requirements.txt` exists (if Python)
- [ ] `assets/` contains any static resources (if needed)
- [ ] `references/` contains links to external docs (if needed)

### Language & Clarity

- [ ] No placeholder text like `[your-file]` or `[command]` remains
- [ ] All instructions are concrete and specific
- [ ] Sentences are short and direct (1-2 clauses max)
- [ ] No jargon without explanation
- [ ] British or American English used consistently

### Final Review

- [ ] I have read the entire SKILL.md as if I were a new user
- [ ] Every instruction is clear and unambiguous
- [ ] I would be able to follow this guide without contacting support
- [ ] The Last Updated date is today's date
- [ ] The Status field is set to "Active"

## Common Mistakes to Avoid

❌ **Mistake 1**: Vague descriptions  
✅ **Fix**: Be specific. "Convert files" → "Convert Markdown (.md) files to formatted PDF with embedded fonts and images"

❌ **Mistake 2**: Placeholder commands  
✅ **Fix**: Use real, tested commands. "`python script.py [input]`" → "`python scripts/markdown_to_pdf.py README.md`"

❌ **Mistake 3**: Missing prerequisites for one OS  
✅ **Fix**: Document installation for macOS, Ubuntu/Debian, and Windows

❌ **Mistake 4**: No expected output shown  
✅ **Fix**: Show exactly what the user will see: the success message, file creation, terminal output

❌ **Mistake 5**: Untested commands or scripts  
✅ **Fix**: Test every command and script locally before adding to documentation

❌ **Mistake 6**: Too many features in one skill  
✅ **Fix**: If "Procedure" has more than 5 steps, consider splitting into multiple skills

## Quick Checklist for Submission

```bash
# Before committing, run this checklist:
cd skills/my-skill-name

# 1. Check SKILL.md exists
[ -f SKILL.md ] && echo "✅ SKILL.md exists" || echo "❌ SKILL.md missing"

# 2. Check scripts exist
[ -d scripts ] && echo "✅ scripts/ exists" || echo "❌ scripts/ missing"

# 3. If Python, check requirements.txt
[ -f scripts/requirements.txt ] && echo "✅ requirements.txt exists" || echo "❌ requirements.txt missing"

# 4. Count lines in SKILL.md (should be 150+)
wc -l SKILL.md

# 5. Check for placeholder text
grep -i "\[your-\|\[example\|\[command\|TODO\|FIXME" SKILL.md && echo "❌ Placeholder text found" || echo "✅ No placeholders"
```
